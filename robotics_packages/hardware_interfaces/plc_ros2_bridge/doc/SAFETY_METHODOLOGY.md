# Safety Validation Methodology for ROS 2 ↔ PLC Integration

**Author:** Seyi R. Afolayan  
**Date:** 2025  
**License:** MIT

## Scope

This document defines the safety validation methodology for integrating ROS 2-based
adaptive robotics with PLC-governed industrial control systems. It establishes the
principles, responsibilities, and validation requirements that govern the boundary
between these two control domains.

This methodology applies to manufacturing deployments where:
- A PLC is the safety controller of record (per IEC 62061 / ISO 13849)
- ROS 2 provides adaptive control capabilities (path planning, force control, etc.)
- The `plc_ros2_bridge` package mediates communication between the two systems

## Fundamental Principle

**The PLC retains ultimate safety authority at all times.**

The ROS 2 layer operates as a **supervisory setpoint generator**. It computes desired
robot behavior and submits requests to the PLC. The PLC independently validates these
requests against safety constraints before executing them. If communication with ROS 2
is lost, the PLC autonomously transitions to a safe state.

This architecture means that a failure in the ROS 2 layer — including software bugs,
network latency, or hardware faults — cannot create an unsafe condition, because the
PLC's safety logic operates independently on certified deterministic hardware.

## Responsibility Matrix

| Function | ROS 2 Layer | PLC Layer |
|----------|------------|-----------|
| Adaptive path planning | **Primary** | — |
| Setpoint generation | **Primary** | Receives & validates |
| Joint velocity/position limits | Pre-validates (advisory) | **Enforces** (authoritative) |
| Emergency stop | Monitors & responds | **Executes** |
| Safety zone monitoring | — | **Executes** |
| Light curtain response | — | **Executes** |
| Communication watchdog | Sends heartbeat | **Monitors & enforces** |
| Drive enable/disable | Requests | **Controls** |
| Fault recovery | Requests reset | **Requires operator confirmation** |

## Validation Requirements

### V1: Pre-Deployment Tag Validation

Before the bridge enters OPERATIONAL state, it must verify that all required PLC tags
exist and are accessible. This catches configuration mismatches at startup rather than
during operation.

**Requirement:** `PLCHardwareInterface::on_activate()` calls `PLCInterface::validateTags()`
for all tags in the `TagRegistry`. Activation fails if any required tag is missing.

### V2: Command Range Validation

Every velocity and position command generated by the ROS 2 controller must be validated
against configured safety limits before being written to the PLC.

**Requirement:** `SafetyValidator::validateJointVelocities()` is called every control
cycle. Commands exceeding limits are zeroed, not clamped — clamping could create
unpredictable robot behavior.

### V3: State-Gated Writes

Command writes to the PLC are only permitted when the `SafetyValidator` is in the
OPERATIONAL state. In all other states (IDLE, ESTOP, FAULT, RESET_REQUIRED), writes
are blocked and zero commands are sent.

**Requirement:** `SafetyValidator::validateWrite()` checks the current safety state
before every write operation.

### V4: Watchdog Integrity

Communication loss between ROS 2 and the PLC must be detected within a bounded time
and result in a safe stop.

**Requirement:** 
- ROS 2 increments a heartbeat tag every control cycle
- PLC monitors the heartbeat and triggers safe stop if no update within timeout
- ROS 2 monitors the PLC's watchdog status tag and enters FAULT if unhealthy
- Timeout is configurable but defaults to 500ms

### V5: Graceful Degradation

If the PLC connection is lost during operation, the bridge must:
1. Immediately stop sending commands (zero all outputs locally)
2. Transition the SafetyValidator to FAULT state
3. Attempt reconnection with configurable backoff
4. Require explicit operator acknowledgment before resuming OPERATIONAL state

**Requirement:** `PLCHardwareInterface::write()` checks `PLCInterface::isConnected()`
before every write. Connection loss triggers the FAULT → RESET_REQUIRED → IDLE path.

## PLC-Side Requirements

For this safety methodology to be effective, the PLC program must implement:

1. **Independent velocity limiting:** The PLC must independently clamp or reject
   velocity commands that exceed safe limits, regardless of what the ROS 2 layer sends.

2. **Watchdog monitor:** The PLC must monitor the ROS 2 heartbeat tag and
   autonomously zero robot outputs if the heartbeat stops updating.

3. **Safety-rated I/O processing:** E-stop circuits, light curtains, and safety
   zone monitors must be handled by the PLC's safety task, which runs independently
   of the main program task.

4. **Enable/disable handshake:** The PLC must require an explicit enable sequence
   (typically initiated by a physical operator control) before accepting ROS 2
   setpoints. Power-on does not imply control authority.

## Compliance Notes

This methodology does not constitute a formal safety assessment. Deployments must
be evaluated by qualified safety engineers per the applicable standards:

- **IEC 62061:** Safety of machinery — Functional safety of electrical/electronic
  control systems
- **ISO 13849-1:** Safety of machinery — Safety-related parts of control systems
- **IEC 61508:** Functional safety of E/E/PE safety-related systems
- **ANSI/RIA 15.06 (now R15.806):** Industrial robot safety requirements (U.S.)

The `SafetyValidator` in this package provides a **software defense-in-depth layer**.
It supplements but does not replace PLC-based safety functions running on SIL-rated
hardware.
